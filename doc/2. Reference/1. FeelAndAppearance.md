느낌과 모양새 
===========

이 문서는 구현을 위한 설계문서로써 제품의 모양새와 느낌을 정의하고 설명함

- 이 문서의 상위에는 개념설계서가 있음
- 이 문서는 모바일 성능최적화 문서와 연관관계가 있음
- 이 문서의 하부에는 위젯 화면설계서가 있음


전제
---
모양새와 느낌을 정의하기 위한 전제

### 사업특성에 관한 전제
- 일반 소비자와 내부 구성원 모두에게 강력한 브랜딩, 마켓팅, 디자인, 커뮤니케이션능력이 있다고 인지됨으로 SKTelecom에 걸맞는 미려함이 있어야함
- 모바일에 방점을 가진 텔레커뮤니케이션 사업자로써 본 제품을 만듬으로 제품의 모양새와 느낌이 모바일과 연관 깊어야함

### 제품특성에 관한 전제
- 모바일에 방점을 가진 제품임으로 제품의 모양새와 느낌이 모바일에 특징을 가져야함
- 기본적으로는 내부 서비스를 향후 외부공개를 통한 확산을 고려하고 있기 때문에 다양한 종류의 내용과 결합이 가능하도록 해야함
- 안드로이드, iOS의 이종간 플랫폼을 지원함으로 이질감 없이 결합이 가능하도록 해야함 
- 트위터 부트스트랩의 방식에 기반함

### 시장환경에 관한 전제
- 웹디자인도 모바일앱의 영향을 강력하게 받고 있으며 우리 제품도 이러한 경향을 수용함
- 터치를 활용한/주안한 웹 디자인이 커다란 디자인 경향을 만들고 있으며 이러한 경향을 수용함


방향
----
전제로부터 출발한 전체적 방향과 개념

- 만져질것 같은 느낌의 극대화
- 안드로이드/아이폰/다양한 내외부 서비스에 녹아드는 중성적 느낌
- 자극에 반응하는 섬세한 느낌의 극대화


개념
----
만지고 즐기는 **Tangeble, Pervasive, Reactive** 디자인 


구현
---
방향과 개념을 토대로 어떻게 구현할 것인지 설명. 모양과 느낌을 설계하는 디자이너, 모양과 느낌을 구현하는 퍼블리셔, 기능을 구현하는 개발자가 아래 내용을 검토하여 실작을 하도록 함

### 구현방안
방향과 개념에 따른 구체적인 방안으로 [트위터 부트스트랩][BtS]을 기본으로하여 [부트스트랩의 사용법][BtS:설명]을 그대로 이어받아 부트스트랩의 장점을 잘 활용할 수 있도록 함

1. **Tangeble** 만져질것 같은 느낌의 극대화  
손으로 직접 화면을 만지는 모바일/패드 단말의 장점을 살림
	
	- 질감의 적극적 활용
	- 양각/음각의 활용
	- 터치 입력을 위한 적절한 크기  

1. **Pervasive** 단말/내용에 녹아드는 중성적 느낌  
iOS, 안드로이드등 단말 종류나 웹앱으로써 담고있는 서비스, 기능의 내용과 무리가 없이 어울리고 순응하는 느낌을 제공
 	
 	- 무채색 색상의 적극적 활용
 	- 강조부분에서 색상의 극적 대비 활용
 	- 약간 둥근 모서리를 가진 박스 활용   

1. **Reactive** 자극에 반응하는 섬세한 느낌의 극대화  
손으로 직접 화면을 만질때 가볍고 작은 애니메이션을 삽입하여 살아있고 작동하는 서비스를 제공하는 인상을 제공
 	
 	- 작은느낌을 전달하는 가벼운 애니메이션 활용

### 요소별 구현방안 및 범위
요소별 구현방안과 범위. 느낌과 모양새를 구현하기 위해서는 단순히 디자인이 필요한게 아니라 개발과 스타일링 같이 기능과 밀접한 연관이 있으며 여기서 부터 서술된 내용은 기능을 구현하는 개발자, 느낌과 모양새를 표현하는 디자이너, 표현된 느낌과 모양새를 코드로 구현하는 퍼블리셔가 함께 검토 및 구현을 하도록 함

#### 테마
제공되는 느낌과 모양세의 커다란 범주를 테마라고 하며 3가지를 테마를 제공함 

1. **Conor Stone** 기본테마
	- 구현방향을 가장 잘 살린 테마로 기본적으로 제공

1. **Booted** 부트스트랩 모바일 확장
	- 부트스트랩의 모바일 확장 테마

1. **WireFrame** 백지테마
	- 확장이나 변형을 위해 제공되는 테마
	- CSS구조만 제공

#### 베이스
테마의 전체적인 분위기와 활자와 서체, 배치와 격자구조를 포함하며 [부트스트랩 기반구조 _Scaffolding_][Bts > Scaffolding]중 격자와 레이아웃, 반응형 웹을 제외한 나머지와 [부트스트랩의 기본 스타일 _Base CSS_][Bts > Base CSS] 중 폼을 제외한 나머지에 해당함. 베이스는 엘레멘트를 사용할때 별도의 스타일 클래스를 선언할 필요 없이 기본적으로 엘레멘트에 적용되어 있는 스타일임

1. [글꼴](http://twitter.github.com/bootstrap/scaffolding.html#global)  
기본적으로 나눔고딕 [Regular와 나눔고딕 Bold][나눔체], [뫼비우스  Regular와 뫼비우스 Bold][뫼비우스체]를 [모바일에서 최적화][모빌리스 제공 웹폰트]하여 사용

	- _[모빌리스 제공 웹폰트]의 나눔글꼴처럼 뫼비우스글꼴의 용량을 줄여야 함_
	- _[모빌리스 제공 웹폰트]의 나눔글꼴을 그대로 사용하려면 [모빌리스][모빌리스 제공 웹폰트]측에 라이센스 확인이 필요함_
	- _나눔고딕은 변경후 배포가 가능하나 뫼비우스체는 별도의 라이센스 검토가 필요함_

#### [폼][Bts > BaseCSS > Forms]
BtS의 BaseCSS에 해당하나 본 사업에서는 별도의 항목으로 분리토록함. 기본적인 개요는 본 문서의 `베이스`를 참고 하도록 함. 아래의 구현, 기능, 느낌과 모양새에 관련한 방안과 범위는 테마중 Conor Stone을 기준으로 함

##### 공통
폼에 관련된 공통적인 요소

###### 상태
폼의 상태를 표현하는 클래스 모음

- 구현 : BtS에 있는것을 스타일 확장 
- 기능 : 기능없음
- 느낌과 모양새 : 방향성에 맞추어 모바일에 알맞게 적용

- 내용 : input 엘레멘트에 해당하는 상태   

	- Focused input  
	현재 선택되어 커서가 위치한 상태  
	
	- Uneditable input  
	일반적으로 미리 값이 들어가 있어 편집이 안되는 상태로 보통 이곳에 들어있는 값은 함께 전송이 되는 상태  
	
	- Disabled input  
	일반적으로 값이 부여되지 않고 편집, 입력이 안되는 상태로 보통 전송시 누락되는 상태  
	
	- Input with warning  
	보통 입력값의 유효성검사를 통과하지 못하였을때 상태 
	
	- Input with error
	보통 입력값이 유효성 검사를 통과하지 못하거나 값 자체에 이상이 있는 상태
	
	- Input with success
	보통 입력값이 유효성 검사를 통과하여 적절한 값일때의 상태

- 내용 : checkbox 엘레멘트에 해당하는 상태
	
	- Disabled checkbox  
	일반적으로 값이 부여되지 않고 편집, 입력이 안되는 상태로 보통 전송시 누락되는 상태

- 내용 : select 엘레멘트에 해당하는 상태
	
	- Select with success  
	보통 입력값이 유효성 검사를 통과하여 적절한 값일때의 상태

	- jQM에서 dialogs는 현재 화면에서 넘어가 전혀 새로운 창을 띄우는 것으로 BtS의 modal과는 다르나 상, 하단과 버튼등 내부 구성이 유사함
	- jQM에서 Popups은 현재 화면에서 내용을 뒤의 배경으로 두어 새로운 창을 띄우는 것으로 BtS의 modal과 유사하나 내부의 담는 내용은 Tool tip, Menu, Nested menu, Form, Dialog, Photo, Scaling images, Map + video iframes, Overlay panels등 내용위에 겹치는 모든 형식을 Popup이라 하며 이중 Dialog이 Bts의 modal에 해당함
	- 금번구현에서는 기능은 BtS기준에 맞추어 진행함
